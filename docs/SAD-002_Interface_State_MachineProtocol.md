1. Document Purpose

This document defines the data contracts and state transitions that govern the Sovereign Skein. Because the system runs in a decentralized, asynchronous loop (Cron pulses + Telegram interrupts), it relies on a strict State Machine to prevent race conditions (e.g., trying to execute a draft that isn't finished).

2. The State Machine Lifecycle

Every Strategic Merit Contract (SMC) identified by the system follows a specific lifecycle. The status column in the Ledger dictates which Node (Assessor or Executor) is allowed to touch the target.

        PENDING --> REJECTED : AI Triage (Meatbag/No Yield)

    state Drafting {
        PENDING --> DRAFT_REQUESTED : Director runs /draft

        DRAFT_READY --> POST_REQUESTED : Director runs /post
        POST_REQUESTED --> COMPLETED : PR Opened / Comment Posted
        POST_REQUESTED --> ERROR : API / OPSEC Failure
        POST_REQUESTED --> AMEND_REQUESTED : OPSEC Red-Line Triggered
    }

        VULTURE_PENDING --> VULTURE_AWAITING_CMD : Assessor notifies Telegram
        VULTURE_AWAITING_CMD --> VULTURE_AUTHORIZED : Director runs /vulture
        VULTURE_AUTHORIZED --> COMPLETED : Vulture script executes
    }

3. Data Schemas

The system uses an Index-Sidecar model to prevent CSV bloat and file corruption.

3.1 The Ledger Index (database/skein_index.csv)

This is the high-speed state tracker. It is read by the HUD and written to by all nodes.

id: String (e.g., T1006). The unique GitHub Issue ID prefixed with 'T'.

status: String. Must strictly match the State Definitions above.

timestamp: String (YYYY-MM-DD HH:MM:SS). Time of first discovery.

title: String. The title of the GitHub Issue.

url: String. The URL to the GitHub Issue.

payout_est: String (e.g., 25 RTC, 10 USDC, TBD). Extracted for HUD display.

3.2 The Vault Sidecar (vault/T{ID}/intel.json)

This is the deep-context file used strictly by the AI "Brain" for reasoning.

{
  "id": "T1006",
  "title": "[BOUNTY] Fix authentication bug",
  "url": "[https://github.com/owner/repo/issues/1006](https://github.com/owner/repo/issues/1006)",
  "body": "Full markdown body of the issue here...",
  "created_at": "2026-03-08T15:00:00Z",
  "vulture_target_user": "maintainer_handle",
  "skein_status": "RETAINED", 
  "reason": "DECISION: PENDING, REASON: Target offers financial reward and requires Python fix."
}


Note: skein_status in the sidecar tracks the AI's internal reasoning (RETAINED vs REJECTED) so the Assessor doesn't waste tokens re-triaging the same issue across multiple pulses.

3.3 The Active Payload (vault/T{ID}/draft_payload.md)

Generated by the Executor. This is raw Markdown text. It contains the exact string that will be posted to GitHub, or the code blocks that will be committed to a Pull Request.

3.4 The Memory Log (logs/self_learning.jsonl)

A JSON-Lines file used to track Director feedback. The Assessor and Executor parse this to prevent repeating past mistakes.

{"timestamp": "2026-03-08", "target_id": "T999", "type": "DIRECTOR_AMENDMENT", "lesson": "Never use the word 'AI' in the payload."}
